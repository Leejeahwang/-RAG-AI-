## 🔄 PC → 라즈베리파이 전환 시점 가이드

> [!IMPORTANT]
> **Phase 2 끝 (5주차)이 최적의 전환 시점입니다.**

### 왜 PC에서 최대한 완성해야 하는가?

| 이유 | 설명 |
|------|------|
| **개발 속도** | PC에서 코드 수정→테스트 주기가 RPi보다 5~10배 빠름 |
| **디버깅 용이** | 에러 발생 시 원인이 코드인지 하드웨어인지 구분 가능 |
| **모델 실험** | 다양한 모델·프롬프트를 빠르게 실험 가능 |
| **팀 협업** | 모든 팀원이 각자 PC에서 병렬 개발 가능 |

### PC에서 반드시 완료해야 할 것
1. ✅ 전체 파이프라인이 **터미널에서 끝까지 동작** (카메라→분석→검색→출력)
2. ✅ 실제 PDF 데이터 기반 RAG가 정확한 결과 반환
3. ✅ STT/TTS가 PC에서 동작 확인
4. ✅ GUI 프로토타입 완성

### 라즈베리파이에서만 해야 할 것
- GPIO, CSI 카메라 등 **RPi 전용 하드웨어** 연동
- **실제 성능 측정** (응답 시간, 메모리 사용량)
- **경량화 튜닝** (모델 교체, 양자화 레벨 조정)
- 부팅 자동 시작, 전력 관리

### 조기 확인 추천 (Phase 1 중 1회)
> [!TIP]
> **1주차에 재황님이 RPi에 Ollama만 설치해서 LLaVA 추론 속도를 한번 측정해보세요.** 이 수치가 전체 프로젝트의 제약 조건을 결정합니다. 만약 너무 느리면 더 가벼운 모델(`moondream2`, `phi-3-vision`)로 일찍 전환해야 합니다.

## 🔀 Git 협업 규칙 (충돌 방지 가이드)

4명이 동시에 개발할 때 코드가 꼬이는 것을 막기 위해 **단순화된 Git Flow**를 사용합니다.

### 1. 브랜치 전략
- `main` 브랜치: 항상 동작하는 안정적인 배포용 코드 (건드리지 않음)
- `dev` 브랜치: 통합 테스트용 메인 개발 브랜치. 기능이 완성되면 여기로 합칩니다.
- `feature/{기능명}` 브랜치: 각자 작업하는 개인 브랜치

**생성 예시:**
- 승훈: `feature/gui-app`
- 규태: `feature/vision-camera`
- 종화: `feature/pdf-data`
- 재황: `feature/docker-setup`

### 2. 작업 순서 (매일 할 일)
1. **아침에 최신화**:
   ```bash
   git checkout dev
   git pull origin dev
   ```
2. **내 브랜치 만들고 이동 (또는 기존 브랜치 업데이트)**:
   ```bash
   git checkout -b feature/내기능
   # 만약 dev에 남이 올린 코드가 추가됐다면 내 브랜치로 가져오기
   git merge dev
   ```
3. **작업 후 커밋 & 푸시**:
   ```bash
   git add .
   git commit -m "feat: 카메리 캡처 기능 추가"
   git push origin feature/내기능
   ```

### 3. 코드 합치기 (Pull Request & Merge)
- 내 로컬에서 바로 `dev`로 `git merge` 하지 않습니다.
- Github에서 `feature/내기능` → `dev`로 **Pull Request(PR)**를 생성합니다.
- (권장) 리뷰어 1명 이상이 확인한 후, 충돌이 없으면 `dev`로 머지합니다.
- 충돌(Conflict)이 발생하면 Github UI에서 충돌난 파일 수정 후 머지합니다.

### 4. 구조화(리팩토링)의 힘
이미 Phase 1에서 코드를 `vision/`, `rag/`, `config.py` 등으로 완전히 분리해 두었습니다.
**규태님은 `vision/` 폴더만 만지고, 승훈님은 `rag/` 폴더만 만지기 때문에 애초에 git 충돌이 발생할 확률이 매우 낮습니다.** (충돌은 같은 파일의 같은 줄을 수정할 때만 발생합니다.)
